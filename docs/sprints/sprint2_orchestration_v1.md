# Sprint 2 (недели 3-4): Orchestration v1

**Статус:** в работе · scope и приемка: [../sprint2_scope.md](../sprint2_scope.md)

## Цель спринта

Собрать стабильный E2E-контур для **1 целевого сценария (склад + манипулятор)** с базовой оркестрацией и управлением сбоями, с разделением:

- **PyBullet** — быстрые автоматические acceptance-тесты (в т.ч. в Docker).
- **Webots** — визуальный E2E и интеграция с ROS 2 через `webots_ros2` на хосте.

## Объем работ

- Behavior Tree / state-machine для 1 сценария
- Подключение `ROS2` как транспортного/интеграционного слоя для runtime
- Retry logic, timeout policy, fallback-переходы
- Базовый incident logging
- Replay hooks для разборов инцидентов
- Acceptance-тесты для happy-path и 3 типов сбоев (приоритетно на **PyBullet** + дублирующий прогон в **Webots** при необходимости)

## Технические задачи

- Расширить runtime-оркестрацию до управляемых ветвлений
- Зафиксировать и реализовать ROS2-интерфейсы (`topic/service/action`) для миссии
- Реализовать **sim-адаптеры**: тонкий слой к **PyBullet** (CI) и мост к **Webots** (`webots_ros2`) для демо
- Ввести структурированный формат инцидентов (код, причина, шаг, контекст)
- Добавить replay-идентификатор и сохранение ключевых событий миссии
- Реализовать fallback-переходы в state-machine
- Поднять автоматические acceptance-тесты с детерминированной инъекцией сбоев

## Артефакт спринта

- Воспроизводимый E2E: автотесты на **PyBullet** + демонстрация в **Webots** (склад/манипулятор)

## Definition of Done

- 1 сценарий (склад/манипулятор) выполняется end-to-end в **PyBullet** в штатном режиме (автотесты)
- Тот же сценарий воспроизводим в **Webots** вручную или по скрипту записи (инвесторский контур)
- Связка `orchestration core + ROS2 + sim-адаптер` (см. `docs/simulator_alternatives.md`) без обходных моков на критическом пути
- На 3 типах сбоев система демонстрирует ожидаемую реакцию policy
- По каждому инциденту есть запись для postmortem/replay
- Acceptance-набор запускается одной командой и стабильно проходит

## KPI контроля

- Success rate в sim (целевой порог фиксируется в спринт-плане)
- MTTR для тестовых отказов
- Доля автовосстановлений
- Количество ручных вмешательств на 100 миссий (sim)

## Зависимости

- Готовность Foundation (Sprint 1)
- Зафиксированы среда симуляции и адаптеры (**Webots** + **PyBullet**; см. `docs/simulator_alternatives.md`)
- Базовая установка ROS2 выполнена в Sprint 1
- Базовые telemetry-метрики из v0

## Риски и снижение

- Риск: неустойчивые fallback-ветки  
  Митигирование: table-driven тесты переходов
- Риск: replay не покрывает важный контекст  
  Митигирование: обязательный минимальный набор полей инцидента
- Риск: задержка настройки **Webots + webots_ros2** на хосте  
  Митигирование: сначала закрыть критический путь на **PyBullet** + ROS 2 в Docker; Webots — параллельно для демо
